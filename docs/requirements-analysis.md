## 二、业务规则

### 2.1 层级创建规则
1. **逐级创建**: 必须从一级开始逐级创建，有一级才能选二级，以此类推
2. **层级唯一性**: 同一个项目中，同一级架构层级只能存在一个
3. **最小层级规则**: 选择最小层级后，无法再创建其他层级；任何时候最小层级都可选（空项目可以直接创建最小层级）
4. **完整结构**: 一个完整架构必须有至少一个最小层级

### 2.2 子项关联规则
1. **上级必填**: 非一级子项必须选择一个上级子项
2. **单关联**: 一个子项只能关联一个上级
3. **名称唯一性**: 同一项目同一层级内子项名称不能重复
4. **层级路径**: 子项显示名称需要包含所有上级的完整路径

### 2.3 账号关联规则
1. **最小颗粒度**: 只有最小层级的架构子项才能关联账号
2. **状态判断规则**:
   - `正常`: 该架构下所有账号都正常授权
   - `有失效`: 该架构下至少有一个账号授权失效
   - `待录入`: 该架构下没有任何账号

### 2.4 删除规则
1. **级联删除**: 删除架构层级 → 删除该层级下所有子项 + 删除所有下级架构 → 删除所有下级子项
2. **二次确认**: 删除架构和删除子项都需要二次确认

### 2.5 短链生成规则
1. **独立性**: 每次生成不同参数的短链必须独立，新生成不影响已有短链
2. **缓存复用**: 相同学数参数 → 复用已存在的短链，不重复生成
3. **哈希算法**: 使用用户ID+平台+账号标签+直播监控+直播分析拼接后计算哈希作为唯一ID

### 2.6 数据统计规则
1. **粉丝数量**: 所选日期范围**最后一天**的粉丝数
2. **粉丝增长量**: `最后一天粉丝数 - 第一天粉丝数`
3. **累加规则**: 所有统计数据（播放量、点赞、评论等）都是该架构下**所有账号**的累加

## 三、约束条件

### 3.1 数据约束
| 约束 | 说明 |
|------|------|
| 架构名称唯一性 | 同一项目内架构名称不能重复 |
| 子项名称唯一性 | 同一项目同一层级内子项名称不能重复 |
| 层级唯一性 | 每个层级编号在项目内只能存在一个 |
| 上级引用完整性 | 非一级子项必须存在有效的上级子项 |
| 账号关联约束 | 只有最小层级子项可以关联账号 |

### 3.2 业务流程约束
| 约束 | 说明 |
|------|------|
| 创建顺序约束 | 必须先创建上级架构才能创建下级架构 |
| 删除影响范围 | 删除父级会级联删除所有后代 |
| 层级关闭 | 选择最小层级后无法继续添加新层级 |

### 3.3 UI交互约束
| 约束 | 说明 |
|------|------|
| 下拉选项禁用 | 不可选的层级在下拉框中不能选择 |
| 仅最小层级可选 | 账号关联时下拉只显示最小层级子项 |
| 失效账号排序 | 弹窗中失效账号排在最前面并标红 |

## 四、高风险模块识别

| 模块 | 风险等级 | 风险原因 | 说明 |
|------|----------|----------|------|
| **级联删除** | 🔴 高 | 误删除影响范围大，数据恢复困难 | 删除架构会级联删除所有子项和下级架构，如果误操作会导致大量数据丢失 |
| **短链生成缓存** | 🔴 高 | 涉及哈希算法和缓存逻辑，容易出问题 | 相同参数复用，不同参数独立，缓存一致性需要保证；之前已有bug，修改后容易引入新问题 |
| **层级创建顺序校验** | 🟠 中 | 规则复杂，容易边界场景漏校验 | 各种状态下（删除中间层级后）下拉选项的可用性需要正确处理 |
| **数据统计汇总** | 🟠 中 | 涉及多维度聚合计算 | 粉丝增长量计算、多账号累加容易出错 |
| **授权地址动态维护** | 🟠 中 | 涉及后端配置化改造 | 原有硬编码改为数据表驱动，需要保证拼接正确 |
| **名称唯一性校验** | 🟢 低 | 逻辑简单但边界容易错 | 编辑场景下对重名判断需要排除自身 |
| **权限控制** | 🟠 中 | 跨多个页面，需要保证一致性 | 项目权限判断，不同负责人只能看自己负责架构的数据 |
